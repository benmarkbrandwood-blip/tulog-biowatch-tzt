# MPU-6050 / MPU-6500 Accelerometer — Driver Reference

Board: TZT ESP32-2432S024C (resistive touch variant)
Breakout: generic MPU-6050 or MPU-6500 module (GY-521 footprint)

---

## GPIO Connections

The MPU-6050/6500 uses the shared I²C0 bus alongside the MAX30102.

| Signal | ESP32 GPIO | Direction | Notes |
|--------|-----------|-----------|-------|
| SDA    | GPIO21    | Bidirectional | 10 kΩ external pull-up to 3.3 V required |
| SCL    | GPIO22    | Out        | 10 kΩ external pull-up to 3.3 V required |
| INT    | GPIO25    | In         | Not used in firmware — reads are unconditional at 100 Hz |
| AD0    | GND       | In         | Low = I²C address 0x68; pull to 3.3 V for 0x69 |
| VCC    | 3.3 V     | —          | Module must be supplied from 3.3 V, not 5 V |

> **Pull-up resistors are mandatory.** The ESP32's internal pull-ups (~45 kΩ) are
> insufficient for reliable I²C at 400 kHz. Without 10 kΩ external pull-ups on
> SDA and SCL, `I2C bus is still busy but software timeout detected` errors appear
> in the monitor and the sensor fails to communicate.
>
> **ADC2 / WiFi conflict note**: GPIO21 and GPIO22 are not ADC-capable and are
> therefore unaffected by the classic ESP32's ADC2 / WiFi blockout.

## Chip Identification — WHO_AM_I

The MPU-6050 and MPU-6500 look identical from the driver's perspective: same I²C
address (0x68), same register map, same scale factors. The only firmware-visible
difference is the WHO_AM_I register response.

| Chip | WHO_AM_I (reg 0x75) | Part numbers |
|------|---------------------|--------------|
| MPU-6050 | 0x68 | InvenSense MPU-6050 |
| MPU-6500 | 0x70 | InvenSense MPU-6500, MPU-9250 (MEMS only) |

Both values are accepted by `hal_mpu6050_init()`. The monitor will log which variant
is detected:

```
I hal_mpu6050: WHO_AM_I = 0x70 — MPU-6500/MPU-9250 identified
```

## I²C Bus Configuration

| Parameter | Value |
|-----------|-------|
| Bus       | I²C0 (`I2C_NUM_0`) |
| SDA       | GPIO21 |
| SCL       | GPIO22 |
| Frequency | 400 kHz (`I2C_BUS_FREQ_HZ` in `app_config.h`) |
| I²C address | 0x68 (AD0 low), 0x69 (AD0 high) |
| Timeout   | 50 ms per transaction (`I2C_TIMEOUT_MS` in `hal_bus.c`) |

Bus initialisation and device registration are handled by `hal_bus_init()` in
[main/hal/hal_bus.c](../main/hal/hal_bus.c), which runs before any sensor driver.

## Initialisation Sequence

Performed in `hal_mpu6050_init()` ([main/hal/hal_mpu6050.c](../main/hal/hal_mpu6050.c)).

| Register | Address | Write value | Effect |
|----------|---------|-------------|--------|
| WHO_AM_I | 0x75 | — | Read; verify = 0x68 or 0x70 |
| PWR_MGMT_1 | 0x6B | 0x00 | Clear SLEEP bit — wake device from default sleep state |
| PWR_MGMT_2 | 0x6C | 0x07 | Disable gyroscope axes (STBY_XG/YG/ZG = 1); accelerometer remains active |
| ACCEL_CONFIG | 0x1C | 0x00 | AFS_SEL = 0 → ±2 g full-scale range (16384 LSB/g) |

The driver reads back PWR_MGMT_1 and ACCEL_CONFIG after writing them and logs the
readback to the serial monitor, confirming the writes took effect.

> **Default sleep mode**: both MPU-6050 and MPU-6500 power up with the SLEEP bit
> set in PWR_MGMT_1. Forgetting to clear this bit is the most common cause of
> FIFO / read failures — the accelerometer registers still respond over I²C but
> return the same stale values on every read.

## Accelerometer Data Read

`hal_mpu6050_read_accel()` performs a 6-byte burst read starting at
`ACCEL_XOUT_H` (0x3B). The chip auto-increments the register pointer across all
six bytes in a single I²C transaction.

```
Register  Byte  Content
0x3B      0     ACCEL_XOUT[15:8]
0x3C      1     ACCEL_XOUT[7:0]
0x3D      2     ACCEL_YOUT[15:8]
0x3E      3     ACCEL_YOUT[7:0]
0x3F      4     ACCEL_ZOUT[15:8]
0x40      5     ACCEL_ZOUT[7:0]
```

**Reconstruction:**

```c
*x = (int16_t)((buf[0] << 8) | buf[1]);
*y = (int16_t)((buf[2] << 8) | buf[3]);
*z = (int16_t)((buf[4] << 8) | buf[5]);
```

Each value is a 16-bit two's-complement integer. At ±2 g full-scale (AFS_SEL = 0),
the sensitivity is **16384 LSB/g**.

**Conversion to m/s²:**

```c
accel_x_ms2 = raw_x / 16384.0f × 9.81f   /* g to m/s² */
```

At rest with the sensor flat, the Z axis reads approximately +16384 (1 g upward).
X and Y read near 0.

---

## Scale Factor Reference

| AFS_SEL | Full-scale range | Sensitivity |
|---------|-----------------|-------------|
| 0 (firmware default) | ±2 g | 16384 LSB/g |
| 1 | ±4 g | 8192 LSB/g |
| 2 | ±8 g | 4096 LSB/g |
| 3 | ±16 g | 2048 LSB/g |

The firmware uses ±2 g. Change `accel_cfg = 0x00` in `hal_mpu6050_init()` and
update `HAL_MPU6050_ACCEL_SCALE` in [main/hal/hal_mpu6050.h](../main/hal/hal_mpu6050.h)
if a wider range is needed.

## Firmware Integration

### Acquisition path

`svc_biosignal_acq_step_100hz()` (in [svc_biosignal_acq.c](../main/services/svc_biosignal_acq.c))
calls `hal_mpu6050_read_accel()` every 100 Hz tick and publishes the raw int16_t
values to `biosignal_frame_t.accel_x_raw / accel_y_raw / accel_z_raw` under the
acquisition mutex.

`ecg_sampler_task` (Core 0) reads the frame, stores the int16_t values in
`s_accel_x_scaled / y / z`, and per sample:

1. Writes the raw int16_t value into the 4-second ring buffers `s_accel_x_raw[]`,
   `s_accel_y_raw[]`, `s_accel_z_raw[]` (int32_t, `ECG_WINDOW_SAMPLES` = 400 entries).
2. Tracks per-axis min/max under `s_ecg_spinlock` for chart autoscale.
3. Converts raw → m/s² and writes `rec_row_t.accel_x/y/z` (float) when recording.

All ring buffer writes and min/max updates are protected by `s_ecg_spinlock`
(portMUX — use `portENTER_CRITICAL` / `portEXIT_CRITICAL`, not a FreeRTOS mutex).

### Record screen — Accel tab

The Accel tab (`REC_TAB_ACCEL`) displays all three axes simultaneously as separate
LVGL chart series (white / yellow / cyan). Each series uses a separate
`lv_point_precise_t` array and autoscales independently to fill the chart height.
The chart series are built lazily 800 ms after screen navigation (to avoid an LVGL
9.5 bug with charts containing ≥ 400 points during screen-load animations).

### CSV output

Three float columns written in every row when recording is active:

| Column | Units | Format |
|--------|-------|--------|
| `accel_x_ms2` | m/s² | `%.4f` |
| `accel_y_ms2` | m/s² | `%.4f` |
| `accel_z_ms2` | m/s² | `%.4f` |

Conversion applied at the CSV write site (`ecg_sampler_task`):

```c
row.accel_x = accel_x_loc / 16384.0f * 9.81f;
```

If the sensor is offline (init failed), all three columns write `0.0000`.

## Diagnostic Log Output

At init, the monitor prints:

```
I hal_mpu6050: Init: probing 0x68 on I2C0 (SDA=GPIO21 SCL=GPIO22)
I hal_mpu6050: WHO_AM_I = 0x70 — MPU-6500/MPU-9250 identified
I hal_mpu6050: PWR_MGMT_1 wrote 0x00, readback 0x00 (SLEEP bit clear — awake)
I hal_mpu6050: ACCEL_CONFIG wrote 0x00, readback 0x00 (AFS_SEL=0, expect 0 for ±2g)
I hal_mpu6050: MPU-6500/MPU-9250 ready — accel ±2g (16384 LSB/g), gyro standby
I svc_biosig: MPU-6050 online — accelerometer valid
```

Once polling, the driver logs a sample every ~1 s:

```
I hal_mpu6050: accel raw X=-243 Y=112 Z=16201  →  -0.146  0.067  9.741 m/s²
```

On init failure, `svc_biosig` logs:

```
W svc_biosig: MPU-6050 offline — accel data not available (init failed)
```
